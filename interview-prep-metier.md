## D. Assignment 1 — สรุปแนวทาง (เวอร์ชันพูด 1 นาที)

> โจทย์คือทีมดูแลหลายโปรเจกต์ที่ feature กับ UI คล้ายกัน ต่างกันแค่ business logic บางส่วนกับ design token แต่ทุกครั้งที่เริ่มใหม่ต้องตั้งค่าและตรวจสอบซ้ำ ผมเสนอ 3 ชั้นครับ
>
> ชั้นแรกคือ **Starter Kit** ทำโครงโปรเจกต์กับ config มาตรฐานไว้พร้อมใช้ ทั้ง ESLint, Prettier, TypeScript เริ่มโปรเจกต์ใหม่แค่ clone แล้วปรับเฉพาะส่วนที่ต่าง ไม่ต้องตั้งค่าใหม่ทุกครั้ง
>
> ชั้นที่สองคือ **Design System กับ Design Token** แยกสี ฟอนต์ spacing ของแต่ละแบรนด์ออกมาเป็น token ส่วน component กลางอย่างปุ่ม input การ์ด ทำครั้งเดียว แล้วแต่ละโปรเจกต์ฉีด token ของตัวเองเข้าไป ทำให้ component เดียวกันเปลี่ยนหน้าตาตามแบรนด์ได้โดยไม่ต้องเขียนใหม่
>
> ชั้นที่สามคือ **Monorepo กับ Shared Packages** รวมของกลางอย่าง ui-components, utils, hooks ไว้ที่เดียว แก้ครั้งเดียวทุกโปรเจกต์ได้พร้อมกัน
>
> ภาพรวมคือมันไม่ได้แก้ทุกอย่างในทีเดียว แต่เป็นการลงทุนตั้งแต่แรก เพื่อให้โปรเจกต์ถัดไปเริ่มได้เร็วขึ้น และทีมเอาเวลาไปโฟกัสที่ business logic ของแต่ละงานแทนการ setup ซ้ำครับ

## H. Assignment 2 — validate comment ภาษาไทย 3 ชั้น

> โจทย์กำหนดว่า comment ต้องเป็นภาษาไทยและตัวเลขเท่านั้น ผม validate 3 ชั้นครับ ชั้นแรกที่ frontend เพื่อ UX ให้ผู้ใช้เห็นเตือนทันทีตอนพิมพ์ ชั้นที่สองที่ฝั่ง API เพราะ frontend อย่างเดียวเชื่อไม่ได้ ใครยิง request ตรงเข้า endpoint โดยข้าม UI ก็ได้ และชั้นที่สามเป็น constraint ที่ database เป็นด่านสุดท้ายกันข้อมูลเสียเข้าไป
>
> ส่วนวิธี validate ผมใช้ regex เช็คช่วง Unicode ภาษาไทย คือ \u0E00 ถึง \u0E7F บวกกับตัวเลขและช่องว่าง ถ้ามีตัวอักษรนอกช่วงนี้ก็ไม่ผ่าน
>
> [ถ้าโดนถามต่อเรื่อง edge case เช่น เลขไทย/อิโมจิ/ขึ้นบรรทัดใหม่ ให้ตอบตามที่ทำจริง ถ้าไม่ได้ครอบก็บอกตรงๆ ว่ายังไม่ได้ครอบ แต่รู้ว่าควรเพิ่ม]
>
> ที่ผมทำหลายชั้นเพราะตอนฝึกงานผมเคยทำ pentest แล้วเห็นบ่อยมากว่า validation ที่ฝั่ง frontend อย่างเดียวโดน bypass ง่าย เลยติดนิสัยกันหลายชั้นครับ

## I. Assignment 2 — schema ของ comment (approve / reject ได้)

> โจทย์ต้องการให้ comment ต้อง approve ก่อนแสดง และต้อง reject อันที่เคย approve ไปแล้วได้ด้วย ผมเลยไม่ใช้ field แบบ boolean approved จริง/เท็จ แต่ใช้ field status ที่มีหลายสถานะ เช่น pending, approved, rejected เพราะมันรองรับการเปลี่ยนสถานะกลับไปกลับมาได้ หน้า public ผมก็ query เฉพาะ status เท่ากับ approved ส่วนหน้า admin เห็นทุกสถานะ แล้วเปลี่ยนได้ตลอด ทำให้ flow ยืดหยุ่นกว่าครับ

## J. Assignment 2 — JWT ใน httpOnly cookie

> ผมเก็บ token ไว้ใน cookie แบบ httpOnly ครับ เหตุผลคือถ้าเก็บใน localStorage แล้วเว็บโดน XSS โค้ดที่ฝังเข้ามาจะอ่าน token ไปได้เลย แต่ httpOnly cookie JavaScript อ่านไม่ได้ ก็ลดความเสี่ยงตรงนั้น ส่วนเรื่อง CSRF ที่เป็น trade-off ของการใช้ cookie ผมกันด้วยการตั้ง SameSite เป็น Lax หรือ Strict แล้วก็ตรวจสอบฝั่ง server อีกชั้น เป็นการเลือกป้องกันภัยที่เป็นไปได้มากกว่าครับ

**2. (แสดงว่าอยากโตกับงานจริง)**
> สำหรับคนตำแหน่งนี้ 3 ถึง 6 เดือนแรก ทีมคาดหวังให้ทำอะไรได้บ้างครับ ผมอยากเตรียมตัวให้ตรงกับสิ่งที่ทีมต้องการจริงๆ

**3. (โยงจุดแข็ง automation ของตัวเอง)**
> ตำแหน่งนี้มีโอกาสได้แตะงานฝั่ง workflow automation หรือ integration บ้างไหมครับ พอดีตอนฝึกงานผมทำงานเชื่อม API หลายตัวด้วย n8n มา เลยอยากรู้ว่าทักษะตรงนี้จะได้ใช้ต่อที่นี่ไหม

**4. (แสดงว่าแคร์คุณภาพและทีม ไม่ใช่แค่เขียนโค้ดให้เสร็จ)**
> กระบวนการ code review กับ deploy ของทีมเป็นยังไงครับ ผมอยากเข้าใจว่าทีมรักษาคุณภาพงานข้ามหลายโปรเจกต์กันยังไง

**5. (แสดงว่าอยากอยู่ยาวและพัฒนา)**
> บริษัทมีแนวทางสนับสนุนให้ dev ได้เรียนรู้ของใหม่ยังไงบ้างครับ ผมเห็นว่าบริษัทเริ่มทำงานฝั่ง AI ด้วย เลยอยากรู้ว่าทีมเปิดให้ลองเทคโนโลยีใหม่ๆ มากแค่ไหน

**6. (จริงใจและแสดงว่าให้เกียรติคนสัมภาษณ์)**
> ส่วนตัวคุณเองที่ทำงานที่นี่ อะไรคือสิ่งที่ทำให้อยากอยู่ทำงานที่ Metier ต่อครับ — ผมอยากได้ยินมุมจากคนในจริงๆ
